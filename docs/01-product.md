# 产品细节

## 产品定位

GPUFleet 是一个轻量 GPU 资源观察和服务端运维平台，用于把分散在不同网络环境中的 NVIDIA GPU 设备聚合到一个公网 Web 面板中。它不是任务调度器，也不是远程控制系统；它更像一个能跨 NAT、跨地点运行的 GPU Fleet 看板，帮助管理员看清设备在线状态、GPU 负载变化、显存和温度走势、当前进程占用、数据留存、更新状态和服务健康。

它关注三类问题：

- 当前有哪些 GPU 在线，它们是否健康。
- 每张 GPU 的利用率、显存、温度、功耗、进程占用如何变化。
- 一段时间内 GPU 使用率、空闲率、峰值、异常次数如何统计。
- 过去 24H、7D、30D 的功耗是否可解释，哪些 GPU 存在高温、限速或空转高耗。

## 目标用户

- 小型实验室、个人工作站集群、训练机器池。
- 不希望搭建完整 Prometheus/Grafana/Exporter 体系的团队。
- 设备分布在家庭宽带、办公室、云主机、远程机房等不同网络下的用户。

## 核心约束

- 服务端位于公网，可被客户端访问。
- 客户端可能位于 NAT、动态 IP、内网或临时网络。
- 服务端不能主动访问客户端。
- 服务端不能对客户端产生配置影响或执行影响。
- 依赖尽可能少，默认单机部署即可运行。
- 数据库必须控制磁盘占用，默认至少预留 800MiB 空闲空间。

## 已覆盖功能范围

### 设备总览

- 在线设备数。
- 离线设备数。
- GPU 总数。
- 总显存、已用显存。
- 当前平均 GPU 利用率。
- 高温 GPU 数。
- 最近 24 小时异常事件数。

### 设备列表

- 设备别名。
- 操作系统。
- Agent 版本。
- GPU 数量。
- 最近上报时间。
- 在线/离线状态。
- 最近错误摘要。

### GPU 详情

- GPU 型号。
- GPU UUID 或脱敏 ID。
- 驱动版本。
- CUDA 版本，后续通过 NVML 或驱动侧信息补齐；当前 `nvidia-smi --query-gpu` 本机不可直接查询该字段。
- VBIOS、Compute 能力、显示状态、驱动模型。
- 显存总量。
- GPU 利用率曲线。
- 显存使用曲线。
- 温度曲线。
- 功耗曲线。
- 风扇转速、图形/显存/SM/视频时钟、P-State、PCIe 当前和最大链路信息。
- ECC/MIG 状态，若设备支持。

### 进程视图

- 当前占用 GPU 的进程。
- PID。
- 进程名。
- 占用显存。
- 可选用户名。
- 可选命令行摘要。

默认不采集用户名和命令行，避免泄露敏感信息。

### 统计记录

- 按设备、GPU、时间范围统计平均利用率。
- 统计峰值温度、峰值功耗、峰值显存。
- 统计空闲时长和空闲占比。
- 统计离线次数和离线时长。
- 支持 1H、6H、24H、7D 和 30D 范围，长范围查询使用 rollup 索引和边界容错。

### 能耗与热状态

- 根据 GPU 上报的当前功率、温度、利用率和时钟限速原因生成只读能源摘要。
- 支持 24H、7D 和 30D 耗电量估算、电费估算、平均/峰值功率和采样覆盖率。
- 按 GPU 展示能耗排行、峰值温度、空转高耗、限速状态和诊断项。
- 设备离线或采样缺口过大时不继续累计能耗，避免把离线时间误算成持续耗电。
- 设置页可配置电价、货币、高温阈值、空转利用率阈值和空转功率阈值；这些参数只影响展示估算和诊断口径。
- 能耗板块不提供功耗墙、风扇、频率、任务暂停或其他客户端控制能力。

### 服务运维

- 首次启动配置语言、密码、端口和可选 HTTPS 证书。
- 设置页修改密码、语言、端口、证书、磁盘预留空间、自动更新开关和更新代理。
- 数据库压缩下载和只读诊断包下载。
- Linux 服务端热备份、冷备份和显式确认恢复脚本。
- 服务端在线更新、供应链来源校验、远端构建、fast-forward 拉取、自动重启和更新完成通知。
- 手动重启服务，前端全屏等待恢复。
- 访客模式、访客脱敏总览、访客曲线和访客记录。

### 本地事件

默认规则：

- 设备超过 3 个采集周期未上报，标记离线。
- 温度超过阈值，记录高温事件。
- 显存使用率超过阈值，记录显存压力事件。
- 采集失败连续超过阈值，记录 Agent 异常事件。

当前版本会在 Web 上展示健康状态、磁盘状态、审计摘要和诊断包；可配置告警规则和外部通知仍是后续增强。

## 非目标

第一版不做：

- 任务调度。
- 训练任务管理。
- 远程 Shell。
- 远程杀进程。
- GPU 设置修改。
- 多租户计费。
- Kubernetes 集群管理。
- 复杂可视化编辑器。

这些功能和“服务端不能影响客户端”的安全边界冲突，后续也应谨慎评估。
